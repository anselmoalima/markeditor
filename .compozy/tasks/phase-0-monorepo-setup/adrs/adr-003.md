# ADR-003: Test stack — Vitest (jsdom) + Playwright + axe-core + MSW

## Status

Accepted

## Date

2026-05-22

## Context

The PRD mandates TDD with ≥80% line / ≥75% branch coverage, unit + integration + type-level + e2e tests, accessibility audits (Axe AA, zero violations), and network mocking for upload flows. The stack must be fast (TDD loop), support React Testing Library, run in CI matrix Node 18/20/22 × React 18/19, and keep e2e isolated from the library package.

## Decision

- **Unit + integration**: Vitest with `environment: 'jsdom'`, `coverage.provider: 'v8'`, setup file importing `@testing-library/jest-dom`. Lives in `packages/markeditor/tests/{unit,integration}`.
- **Type-level**: Vitest `expect-type` (or `tsd`) in `packages/markeditor/tests/type/*.test-d.ts`, run by `tsc --noEmit` against a dedicated `tsconfig.types-test.json`.
- **E2E**: Playwright in `apps/playground/e2e/`, runs against `pnpm --filter playground preview` (built Vite bundle).
- **Accessibility**: `axe-core` + `jest-axe` for unit assertions; `@axe-core/playwright` for e2e.
- **Network mocking**: MSW (`msw`) for image-upload flows.

## Alternatives Considered

### Alternative 1: Jest instead of Vitest

- **Description**: Use Jest with `@testing-library/react`.
- **Pros**: Most established ecosystem.
- **Cons**: ESM support still rough; slower; needs Babel for TS+JSX.
- **Why rejected**: Vitest is faster, native ESM, drop-in API.

### Alternative 2: Cypress instead of Playwright

- **Description**: Use Cypress for e2e.
- **Pros**: Time-travel debugger UX.
- **Cons**: Slower; multi-browser support weaker; CI matrix harder.
- **Why rejected**: Playwright better aligned with PRD’s cross-browser + a11y goals.

## Consequences

### Positive

- Single Vitest config covers unit, integration, type tests, coverage, and a11y.
- Playwright e2e is isolated to playground — does not affect published bundle.

### Negative

- Two test runners to learn (Vitest + Playwright); acceptable trade-off for clear scope separation.

### Risks

- jsdom does not fully emulate Monaco’s WebGL/canvas. Mitigation: Monaco is mocked in unit tests; real Monaco only exercised in Playwright e2e.

## Implementation Notes

- `packages/markeditor/vitest.config.ts` with `coverage: { thresholds: { lines: 80, branches: 75, functions: 80, statements: 80 } }`.
- `apps/playground/playwright.config.ts` with `webServer` running `pnpm preview`.
- One **smoke e2e** mandatory in Fase 0: navigate to `/`, assert a placeholder `<MarkEditor>` mounts without errors — proves the workspace consumes the built library.
