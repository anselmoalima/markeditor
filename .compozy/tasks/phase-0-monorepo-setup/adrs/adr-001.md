# ADR-001: Monorepo with pnpm workspaces + Turborepo

## Status

Accepted

## Date

2026-05-22

## Context

`markmd` ships as an NPM package (`packages/markmd`) but needs a separate, never-published demo app (`apps/playground`) and possibly docs/Storybook (`apps/docs`). Workspaces must share TS/ESLint/Prettier config, install deps once, and let the playground consume the library via symlink without a publish round-trip. CI must avoid rebuilding/retesting untouched workspaces.

## Decision

Adopt **pnpm workspaces** for dependency hoisting and symlinked workspace deps, and **Turborepo** (`turbo.json`) for task orchestration with content-hashed local caching. Workspace globs: `packages/*`, `apps/*`. Playground depends on the library via `"markmd": "workspace:*"`.

## Alternatives Considered

### Alternative 1: npm workspaces + Nx

- **Description**: Use stock npm workspaces with Nx as the build orchestrator.
- **Pros**: No third-party package manager; Nx has richer plugin ecosystem.
- **Cons**: npm install is slower; Nx config is heavier than `turbo.json` for a 2-package repo.
- **Why rejected**: Overhead unjustified for current scope; pnpm is the de-facto OSS choice.

### Alternative 2: Single package (no monorepo)

- **Description**: Keep playground inside the published package under `examples/` or `dev/`.
- **Pros**: Simplest setup.
- **Cons**: Demo code risks leaking into the tarball; demo-only deps (Vite, React Router) pollute peer/dev deps; harder to add docs app later.
- **Why rejected**: PRD §2 mandates publish/demo separation.

### Alternative 3: Yarn Berry workspaces

- **Description**: Yarn 4 PnP or node_modules linker with workspaces.
- **Pros**: Mature workspace support.
- **Cons**: PnP breaks Monaco/Vite tooling in edge cases; smaller OSS adoption in 2026.
- **Why rejected**: pnpm has fewer ecosystem footguns for this stack.

## Consequences

### Positive

- Fast installs and disk-efficient symlinks.
- Turbo cache skips unchanged tasks → fast PR feedback.
- Clear publish boundary: only `packages/markmd` is published.

### Negative

- Contributors must use pnpm; npm/yarn installs are blocked by the `packageManager` field.

### Risks

- Turbo cache invalidation bugs can hide stale outputs. Mitigation: enforce `turbo run build --force` in release workflow.

## Implementation Notes

- Pin pnpm via Corepack: `"packageManager": "pnpm@9.x.x"` at repo root.
- `pnpm-workspace.yaml` lists `packages/*` and `apps/*`.
- `turbo.json` defines pipelines for `build`, `test`, `lint`, `typecheck`, `size`, with `dependsOn: ["^build"]` where needed.
