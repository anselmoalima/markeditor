# ADR-006: Lazy-load heavy renderers (KaTeX, Mermaid, highlight.js) via content detection with import cache

## Status

Accepted

## Date

2026-05-24

## Context

PRD RNF-6.5 requires KaTeX, Mermaid, and code-block syntax highlighting to live in separate chunks so a document that uses none of them never pays for them. PRD §6.1 caps the core bundle at <80 KB gzip. Built-in plugins for math/mermaid/code highlighting are active by default (PRD §5.5.4), so opt-in flags from the consumer are not acceptable.

The pipeline must trigger loading exactly when needed, never sooner, and never re-trigger after the chunk is in memory.

## Decision

Each heavy renderer is delivered as a built-in plugin with a two-stage architecture:

1. **Lightweight registration stub** (always in the core bundle): a small object exporting `name`, `remarkPlugins` references that are themselves trivial (a `unist-util-visit`-style probe and a "needs lazy load" marker), and a React component that, when first mounted, triggers `import()` of the heavy library.
2. **Heavy implementation chunk** (lazy): the actual KaTeX/Mermaid/highlight.js code plus its remark/rehype glue.

The pipeline visit pass marks `hast` nodes that require heavy work. The corresponding React component (e.g., `<MathBlock />`, `<MermaidDiagram />`, `<HighlightedCode />`) runs the lazy import on first mount and stores the resolved module in a module-level singleton (`LazyRegistry`). Subsequent mounts use the cached module synchronously.

Content detection is a single regex pass over the raw markdown before parsing — `O(n)` and runs in <2 ms on 10k-line documents — that pre-warms the cache: if `$` or `\\(` is seen, KaTeX `import()` fires; if a fenced ` ```mermaid` block is seen, Mermaid `import()` fires; if any fenced block with a language tag is seen, highlight.js fires.

`LazyRegistry` exposes `prime(name)` for prefetching and `get(name): Promise<Module>` for components.

## Alternatives Considered

### Alternative 1: Suspense-only loading at the React boundary

- **Description**: No content detection; each `<MathBlock />` triggers `import()` and suspends until the module resolves.
- **Pros**: Simplest implementation.
- **Cons**: First preview render of a math-heavy document shows a long suspense fallback; subsequent renders are fine. Bad first-paint UX. Breaks SSR snapshots (math nodes render as fallback).
- **Why rejected**: First-paint UX matters; prefetching from markdown text is cheap.

### Alternative 2: Eager opt-in props (`enableMath`, `enableMermaid`)

- **Description**: Consumer toggles flags; if false, the plugins don't load.
- **Pros**: Deterministic bundle behavior; no surprise downloads.
- **Cons**: PRD §5.5.4 mandates these plugins are active by default. Opt-in degrades DX significantly.
- **Why rejected**: Contradicts PRD.

### Alternative 3: Single combined chunk for all heavy renderers

- **Description**: Bundle KaTeX + Mermaid + highlight.js into one lazy chunk loaded if any one is needed.
- **Pros**: Fewer network requests.
- **Cons**: A document with only highlighted code blocks pays for KaTeX + Mermaid bytes. Violates separation in PRD §6.5.
- **Why rejected**: Independent chunks honor the budget per feature.

## Consequences

### Positive

- A markdown document with no math, no diagrams, and no fenced code blocks pays nothing beyond the core bundle.
- Detection runs upfront, so by the time the pipeline hits a math node the import is usually in flight or done.
- Cache survives mode toggles, plugin re-registration, and re-renders.

### Negative

- The first document with a heavy feature has a non-zero load delay (chunk fetch). Mitigation: optional `prefetchHeavyPlugins` consumer prop, plus prefetch on Monaco mount (the user is about to type math).
- Content detection regex must be conservative; false positives waste bytes but never break correctness. Acceptable trade-off.

### Risks

- KaTeX's CSS file must also be loaded. Mitigation: dynamic `<link rel="stylesheet">` injection by the lazy module on first call; documented in README.
- Mermaid module is large (~700 KB unminified). Mitigation: only Mermaid's renderer (not the full Mermaid bundle) is dynamically imported; verified via `size-limit` budget per chunk.

## Implementation Notes

- `src/core/lazy/registry.ts` — module-level `Map<string, Promise<Module>>`; `prime`, `get`, `isLoaded`.
- `src/core/lazy/detect.ts` — regex set returning `{ needsMath, needsMermaid, needsHighlight }`.
- Built-in plugins under `src/plugins/builtin/` each export `registerHeavyLoader(registry)` and a stub React component.
- `size-limit` config asserts: core entry <80 KB, math chunk <60 KB, mermaid chunk <250 KB (mermaid is intrinsically large), highlight chunk <40 KB, total with all chunks <500 KB.
