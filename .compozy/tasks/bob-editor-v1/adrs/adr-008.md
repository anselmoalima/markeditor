# ADR-008: Controlled `value` always wins; storage is write-through and restores only when uncontrolled

## Status

Accepted

## Date

2026-05-24

## Context

PRD §5.9 specifies `storage: 'localStorage' | 'sessionStorage' | false`, `autoSave`, `autoSaveInterval`, and supports both controlled (`value` + `onChange`) and uncontrolled (`defaultValue`) usage. When a consumer passes `value` AND `storage`, the interaction must be deterministic. React conventions say "controlled means consumer owns the state"; storage cannot silently override consumer state without violating that.

## Decision

Storage policy:

1. **Mount-time read**: Storage is read **only** if the component is uncontrolled (`value === undefined`) AND `defaultValue === undefined`. The restored value seeds the reducer's initial markdown. If `defaultValue` is provided, storage read is skipped.
2. **Mount-time write**: After mount, every change to `markdown` state (reducer transitions of type `content/setMarkdown`) writes to storage via a debounced effect (`autoSaveInterval`, default 1000 ms). This applies in both controlled and uncontrolled modes — the library never silently throws away the consumer's content; it just records what's currently shown.
3. **Controlled + storage**: Storage records the controlled `value` over time. Useful for "soft backup" cases (consumer reloads page mid-edit → consumer can choose to restore from storage by reading the key on their side). Library does NOT auto-restore; the consumer holds the controlled value.
4. **`onSave` callback**: Independent of storage. Fired on `Ctrl/Cmd+S`. Storage may or may not be enabled.
5. **Dev warning**: If `value !== undefined` and `storage !== false`, log a dev-mode warning explaining that storage is write-only in controlled mode.

## Alternatives Considered

### Alternative 1: Storage takes precedence at mount (overrides `value` once)

- **Description**: On mount, read storage and emit `onChange(restored)` to update the controlled value.
- **Pros**: "Magic" restore for users.
- **Cons**: Violates the React controlled-component contract; consumers expect `value` to be the source of truth; can cause infinite loops if the controlled parent re-renders with the original `value`.
- **Why rejected**: Magic that bypasses controlled state is the worst kind.

### Alternative 2: Storage disabled entirely in controlled mode

- **Description**: If `value !== undefined`, ignore `storage` (no reads, no writes).
- **Pros**: Cleanest separation.
- **Cons**: Loses a useful "soft backup" case where a consumer wants their controlled state mirrored for crash recovery.
- **Why rejected**: Writing is harmless and frequently useful; keeping write semantics costs nothing.

### Alternative 3: Per-prop opt-out (`storageReadOnMount: boolean`)

- **Description**: Expose a flag.
- **Pros**: Explicit.
- **Cons**: Adds another prop; consumers must understand a non-obvious combination.
- **Why rejected**: The "controlled wins" rule covers 100% of cases without a new prop.

## Consequences

### Positive

- Controlled-component contract is preserved.
- Uncontrolled users get the "browser remembers my draft" feature for free.
- Mixed-mode (controlled + storage) is well-defined and useful.

### Negative

- Consumers who *want* mount-time storage restore in controlled mode must implement it themselves (read the key, set their own state). Documented in README "Recipes".
- Two write paths to storage may race in rapid edit sessions. Mitigation: a single debounced effect serialized on `markdown`.

### Risks

- Storage quota exhaustion in browsers. Mitigation: catch `QuotaExceededError`, emit `onError`, disable storage for the rest of the session (with a single dev warning).
- Server-rendered components must skip storage entirely. Mitigation: `typeof window !== 'undefined'` guard inside the storage effect; SSR snapshot uses the props-provided value.

## Implementation Notes

- `src/hooks/useStorageSync.ts` encapsulates read-on-mount-if-uncontrolled + debounced write.
- Status bar reads `savedAt` from reducer state to render "saving..." / "saved 2s ago".
- Storage key: `storageKey` prop (default `markdown-editor-content`). The component generates a per-instance suffix only if `storageKey` is the default and there are multiple instances on the same page (warning logged).
