# ADR-005: Monaco lazy-loaded via dynamic `import()`; SSR/loading fallback is an accessible `<textarea>`

## Status

Accepted

## Date

2026-05-24

## Context

Monaco Editor weighs ~400 KB gzipped and depends on browser globals (`window`, workers). The PRD demands (a) bundle size <80 KB gzip without Monaco (RNF-6.1), (b) Monaco loads <500 ms on 4G (RNF-6.1), (c) the component must remain importable in SSR contexts (Next.js App Router, Remix loaders), (d) markdown editing must remain possible even before Monaco resolves or in SSR snapshots.

## Decision

Wrap the Monaco-backed editor in a `<MonacoEditor />` internal component imported via `React.lazy(() => import('./components/MonacoEditor'))`. The lazy import is guarded by `typeof window !== 'undefined'` and only triggers when `mode === 'edit'` and the consumer has not opted into preview-only.

While Monaco is loading or in any SSR context, the same edit panel renders a styled `<textarea>` (`<TextareaFallback />`) wired into the same reducer:

- Reads `markdown` from state.
- Writes via the same content reducer action used by Monaco's `onDidChangeModelContent`.
- Inherits aria attributes, placeholder, and `readOnly` semantics.
- Receives focus and selection events, dispatching them through the same action shape so `EditorAPI.getSelection` works in fallback mode.
- Is genuinely usable (not a skeleton) — users typing into it produce real `onChange` events that propagate to consumers.

When the dynamic import resolves on the client, Monaco mounts in place of the textarea, the textarea unmounts, and Monaco's model is initialized from the current `markdown` state. Selection is **not** transferred (best-effort; the fallback is short-lived).

## Alternatives Considered

### Alternative 1: Two separate exported components

- **Description**: Export `BobEditor` (client-only) and `BobEditorSSR` (preview-only). Consumer chooses based on context.
- **Pros**: Each export is single-purpose.
- **Cons**: Doubles the public surface, forces consumers to gate via `'use client'` boundaries, breaks the "drop-in" DX in PRD §1.
- **Why rejected**: One component with internal SSR awareness is the documented goal.

### Alternative 2: Render `null` (or a skeleton) on the server, no fallback editor

- **Description**: Detect SSR and render an empty placeholder; only Monaco can edit.
- **Pros**: Smallest bundle path.
- **Cons**: Pre-hydration users can't type; loses graceful degradation when Monaco fails to load (network errors, CSP blocks).
- **Why rejected**: Violates PRD §14 risk mitigation requiring a textarea fallback.

### Alternative 3: Always render textarea, ship Monaco as an opt-in subpath

- **Description**: `bob-editor/monaco` is the upgraded variant; default is textarea-only.
- **Pros**: Bundle is tiny by default.
- **Cons**: Loses the "Monaco-powered" identity in PRD §1. Most consumers want Monaco — they'll just import the subpath, defeating the simplicity gain.
- **Why rejected**: Default DX is the design goal; lazy loading achieves the same bundle budget.

## Consequences

### Positive

- One drop-in component works in SSR and client contexts.
- Bundle without Monaco stays small (Monaco is a lazy chunk).
- Textarea fallback satisfies a11y (semantic `<textarea>` with `aria-label`) and CSP-strict environments.

### Negative

- Brief mode-switch flash possible when Monaco resolves (~200–500 ms on 4G). Mitigation: prefetch Monaco on `mouseenter` of the editor area or on `onMount` if `prefetchMonaco` prop is `true`.
- State sync between Monaco model and reducer must be one-directional (state → model) on Monaco mount to avoid losing keystrokes typed during the swap.

### Risks

- Some bundlers (Next.js Webpack) tree-shake `import()` boundaries poorly. Mitigation: declare `'/* webpackChunkName: "bobmd-monaco" */'` magic comment and document the expected chunk name in README.
- Monaco worker files must be served from the consumer's domain; PRD §1 says "zero-config", but bundlers like Vite may require a worker plugin. Mitigation: README "Bundler recipes" section gives one snippet per bundler (Vite, Next.js, Webpack, Remix).

## Implementation Notes

- Components: `src/components/Editor/MonacoEditor.tsx`, `src/components/Editor/TextareaFallback.tsx`, `src/components/Editor/index.tsx` (orchestrator).
- The orchestrator uses `React.lazy` + `Suspense` with `fallback={<TextareaFallback />}` and `useSyncExternalStore` (or a simple `useState` flag) for SSR detection.
- Prefetch hook: `useMonacoPrefetch({ trigger: 'idle' | 'interaction' | 'mount' })`.
- Monaco mount effect: `editor.setValue(state.markdown)` then attach `editor.onDidChangeModelContent` listener.
