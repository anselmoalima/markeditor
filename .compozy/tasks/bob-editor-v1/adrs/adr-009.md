# ADR-009: TDD pyramid (Vitest unit + RTL integration + Playwright e2e) with `vitest bench` regression gate and Changesets-backed provenance release

## Status

Accepted

## Date

2026-05-24

## Context

PRD §6.7 specifies a comprehensive TDD strategy: unit (Vitest), integration (React Testing Library + user-event), type tests (`expect-type` / `tsd`), e2e (Playwright in `apps/playground`), a11y (jest-axe + @axe-core/playwright), bundle (`size-limit`), package linting (`publint` + `@arethetypeswrong/cli`), and benchmarks (`vitest bench`) with a regression gate at 1.2× baseline (RNF-6.7.6). Coverage targets: ≥80% lines, ≥75% branches. Release strategy (§6.6.5, §18) uses Changesets + GitHub Actions OIDC + `npm publish --provenance --access public`. The CI matrix is Node 18/20/22 × React 18/19.

## Decision

Adopt the full pyramid as specified, with the following structural decisions to make it operable:

### Test layout (inside `packages/bob-editor/`)

- `tests/unit/` — Vitest, fast, jsdom env. Pure functions: pipeline composition helpers, sanitize merge, lazy registry, shortcut parser, plugin manager, debounce, storage hook, FNV hash, theme application.
- `tests/integration/` — Vitest + jsdom + RTL + user-event. The `<BobEditor />` itself, toolbar interactions, modal lifecycle, shortcut firing, mode toggle, controlled-vs-uncontrolled, storage hydration, plugin onMount/onChange/onBeforeParse/onAfterRender flow with cleanup verification.
- `tests/type/` — `*.test-d.ts` files using `expect-type`. Asserts the public types in `src/types.ts` (`BobEditorProps`, `BobEditorRef`, `EditorAPI`, `BobEditorPlugin`, `KeyboardShortcut`, `ToolbarButton`) do not regress.
- `tests/bench/` — `vitest bench`. Fixtures of 1k, 5k, and 10k lines. Each benchmark records `ops/s` to `tests/bench/baseline.json`. CI job `bench` compares against baseline; failure threshold: any metric >1.2× slower than baseline.
- `tests/fixtures/` — markdown samples grouped by feature: `gfm/`, `math/`, `mermaid/`, `alerts/`, `footnotes/`, `code/`, `xss/`, `large/`.
- Playwright lives at `apps/playground/e2e/`. The playground exposes all PRD §7.5 routes; each spec drives one route.

### Coverage and gates

- `vitest run --coverage` produces v8 coverage. CI fails if line coverage <80% or branch coverage <75%.
- `size-limit` runs on PRs; failing PRs are blocked.
- `publint` and `@arethetypeswrong/cli` run as non-failing reporters in PRs (warning surface) and **failing** gates on `prepublishOnly`.
- `jest-axe` assertions inside RTL tests cover modals and primary surfaces; `@axe-core/playwright` covers the e2e flow once per route.

### Bench gate

- Baseline JSON checked into the repo; updated only via dedicated `chore: rebase bench baseline` PRs that link to a perf-affecting change. CI ignores baseline updates on the same PR (a separate "baseline" job runs against the new baseline).

### Release pipeline

- Changesets initialized; PRs that touch `packages/bob-editor/src/**` must include a changeset (CI gate via `changeset status`).
- `release.yml` runs on push to `main`: `pnpm install --frozen-lockfile`, `pnpm -r build`, then `changesets/action@v1` with `publish: pnpm changeset publish` and `NPM_CONFIG_PROVENANCE: true` plus `id-token: write` for OIDC.
- `prepublishOnly`: `build → test → typecheck → lint → publint → attw → size`.
- First publish runs locally with `npm publish --provenance --access public --tag latest` after `npm view bobmd` confirms availability.

## Alternatives Considered

### Alternative 1: Skip bench gate (manual perf checks)

- **Description**: Run `vitest bench` only locally before releases.
- **Why rejected**: Violates RNF-6.7.6.

### Alternative 2: Playwright in `packages/bob-editor` rather than `apps/playground`

- **Description**: Drive Playwright against a fixture page bundled inside the package tests.
- **Cons**: Duplicates the playground app; less realistic; bundlers may produce a different artifact than what consumers receive.
- **Why rejected**: PRD §7.5 explicitly places e2e in the playground.

### Alternative 3: GitHub Actions OIDC vs classic NPM tokens

- **Description**: Use a long-lived NPM_TOKEN secret only.
- **Cons**: No provenance attestation; secret rotation risk.
- **Why rejected**: PRD §6.6.5 mandates provenance via OIDC.

## Consequences

### Positive

- Every category in PRD RNF-6.7.4 has a dedicated test home.
- CI fails loudly on perf regression, bundle bloat, type regression, and a11y violations.
- Releases are reproducible, signed via provenance, gated on every prepublish check.

### Negative

- CI runtime grows with the matrix (3 Node × 2 React = 6 unit-test runs + bench + e2e). Mitigation: Turbo remote cache + selective Playwright sharding.
- Baseline JSON is brittle on shared CI runners. Mitigation: bench is run on a dedicated runner type with consistent hardware; baseline updates require explicit human PR.

### Risks

- Coverage <80% in a feature surface (e.g., obscure plugin paths) blocks merge. Mitigation: feature owners add fixtures before opening the PR; coverage report is posted as a PR comment.
- `attw` flagging is sensitive to `exports` map ordering. Mitigation: lock the map shape per ADR-001 and document the canonical layout in the package README.

## Implementation Notes

- `packages/bob-editor/vitest.config.ts` — jsdom env, setup file with `@testing-library/jest-dom` + `jest-axe/extend-expect`, coverage v8 with thresholds.
- `packages/bob-editor/size-limit.json` — entries per ADR-006 budgets.
- `.github/workflows/ci.yml` — jobs: `lint`, `typecheck`, `test`, `bench`, `size`, `publint`, `attw`. Matrix on `test` only.
- `.github/workflows/release.yml` — Changesets publish.
- `.github/workflows/size.yml` — separate so it can comment on PRs.
- `apps/playground/playwright.config.ts` — browsers chromium, firefox, webkit; sharded by project.
