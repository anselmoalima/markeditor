# ADR-004: GitHub sanitize schema as baseline, plugin-contributed extensions merged at registration

## Status

Accepted

## Date

2026-05-24

## Context

PRD §6.3 mandates XSS sanitization by default with the GitHub schema (`rehype-sanitize`'s `defaultSchema`). Built-in features (math via KaTeX, mermaid SVG, alerts/callouts, slugged headings, code-blocks with copy button) emit HTML/SVG tags and attributes that the GitHub baseline strips (e.g., KaTeX MathML, `class` patterns on `span`, `data-*` attributes on alerts, SVG roots on mermaid). Stripping these would visually break the preview. At the same time, consumers must remain protected against `<script>`, `on*` handlers, `javascript:` URLs, and arbitrary `data:` URLs.

## Decision

Compose the active sanitize schema from layered contributions, applied once per pipeline build:

1. **Baseline**: `defaultSchema` from `rehype-sanitize` (GitHub schema, deep-cloned).
2. **Core extensions**: Hardcoded additions for math (`math`, `mathml`, KaTeX `span`/`semantics`/`annotation`/`mrow` tags + `class` allowlist), mermaid (`svg`, `g`, `path`, `text` with bounded attributes), alerts (`div` + `data-callout-type` + class allowlist), slug (`id` on heading tags).
3. **Plugin extensions**: Each plugin may declare `sanitizeSchema: SchemaExtension` (a partial schema fragment). The plugin manager `merge`s these into the active schema during plugin registration.
4. **User override**: The `sanitize` prop accepts `boolean | Schema`. `false` disables sanitization (with a dev warning). A `Schema` object replaces the merged result entirely. A `(schema) => schema` function receives the merged schema and may extend it.

`rehype-sanitize` runs **before** plugin-supplied `rehypePlugins` that emit React components, so the React render layer can trust the post-sanitize hast. `on*` attributes, `javascript:` and unbounded `data:` URLs are blocked unconditionally (these clauses are not extension-removable; the merger preserves the GitHub blocks even if a user extension would weaken them).

## Alternatives Considered

### Alternative 1: Ship a pre-merged custom schema, no runtime composition

- **Description**: Build one big schema at design time covering math/mermaid/alerts and bake it into the bundle.
- **Pros**: Zero runtime composition cost; deterministic.
- **Cons**: Third-party plugins that introduce new tags (e.g., mentions emitting `<a data-mention>`) cannot extend the schema without consumers passing a full custom schema. Worse DX.
- **Why rejected**: Plugin extensibility is a core selling point.

### Alternative 2: Sanitize is opt-in (`sanitize: false` default)

- **Description**: Trust consumer-provided markdown; let consumers opt in to sanitize.
- **Why rejected**: Violates PRD RNF-6.3 (sanitization active by default).

### Alternative 3: Run sanitize twice — once before plugin transforms, once after

- **Description**: Defensive double-pass to catch plugin-introduced unsafe nodes.
- **Pros**: Belt and suspenders.
- **Cons**: ~2× sanitize cost on large documents (budget-affecting); plugin transforms can interleave React components that the second pass mangles.
- **Why rejected**: Single pass with extension-aware schema is sufficient; defense in depth comes from immutable security clauses, not duplicate passes.

## Consequences

### Positive

- Plugins extend cleanly without forcing consumers to maintain bespoke schemas.
- The hardcoded security clauses (no `on*`, no `javascript:`) cannot be weakened by accident.
- Sanitized output is the same shape that React renders, so the React layer never sees raw user HTML.

### Negative

- Schema merge logic adds complexity. Mitigation: well-tested helper `mergeSanitizeSchema(base, ext)` with property-level union semantics.
- A misconfigured plugin can re-open small XSS vectors. Mitigation: integration tests run OWASP XSS Filter Cheat Sheet vectors against every built-in plugin (PRD RNF-6.7.4).

### Risks

- Mermaid SVG output is wide; allowing too many attributes risks SVG-borne XSS (e.g., `<a href="javascript:">` inside SVG). Mitigation: mermaid plugin renders via React component that takes the rendered SVG and re-sanitizes the SVG string with a strict SVG-only schema before mounting.

## Implementation Notes

- `src/core/sanitize/schema.ts` exports `getCoreSchema()` returning a deep-cloned merged baseline.
- `src/core/sanitize/merge.ts`: helper that unions `tagNames`, `attributes` (per-tag and global), `protocols`, `clobberPrefix`. Asserts that locked clauses (no `on*`, restricted protocols) survive every merge.
- Plugin manager calls `mergeSanitizeSchema(activeSchema, plugin.sanitizeSchema)` in registration order.
- `sanitize` prop function-form: `(merged) => merged` is called once per change in plugins list.
