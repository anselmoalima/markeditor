# ADR-003: Central `useReducer` + Context with imperative `EditorAPI` for plugin extensibility

## Status

Accepted

## Date

2026-05-24

## Context

`<BobEditor />` must support controlled (`value` + `onChange`) and uncontrolled (`defaultValue`) modes, expose imperative methods through a `ref` (`BobEditorRef` â€” `getValue`, `setValue`, `setMode`, `insertText`, etc.), feed an `EditorAPI` to plugins for hook callbacks, and let toolbar buttons and keyboard shortcuts mutate the same state. The plugin contract (`BobEditorPlugin`) requires `onMount(api)`, `onChange(value, api)`, `onBeforeParse`, `onAfterRender`, and lifecycle cleanup. The component must work for many overlapping consumers (toolbar, modals, shortcut manager, plugins, ref) without race conditions or implicit prop drilling.

## Decision

Centralize state in a single `useReducer` inside `<BobEditor />`. State shape: `{ markdown, mode, selection, isLoading, hasError, savedAt, ...flags }`. Wrap the reducer in a `BobEditorContext` (React Context) consumed by `Toolbar`, modals, status bar, and built-in/user plugins via a `useEditorContext()` hook. The Monaco editor instance is held in a `ref` (not in state) and accessed through accessor functions to avoid re-render storms.

`EditorAPI` is an instance object created once per mount via `useMemo`, with method bodies that read live refs and call `dispatch`. The same object is (a) attached to `useImperativeHandle` as the public `ref`, (b) passed to each plugin's `onMount`, and (c) available via `useEditorContext().api`. The reducer is the only source of truth; the Monaco instance is a controlled side-effect synced from state via a dedicated effect.

Controlled-vs-uncontrolled is resolved with a thin wrapper hook (`useControllableState`) that reads `value` if defined, falls back to internal reducer state otherwise, and always calls `onChange` on user-driven mutations.

## Alternatives Considered

### Alternative 1: Zustand store per instance

- **Description**: Internal Zustand store created at mount.
- **Pros**: Less boilerplate; selector-based subscriptions reduce re-renders.
- **Cons**: Adds a runtime dependency not declared in PRD Â§15; tree-shaking less aggressive; harder to reason about lifecycle in tests.
- **Why rejected**: React's `useReducer` + `Context` satisfies the requirement without a new dep.

### Alternative 2: Independent hooks per slice (`useContent`, `useMode`, `useSelection`)

- **Description**: Each piece of state is its own `useState` colocated in `<BobEditor />`.
- **Pros**: Maximum modularity.
- **Cons**: Cross-slice updates (e.g., a toolbar action that mutates content AND focuses the editor) require coordinated effects; plugins must subscribe to many hooks; race conditions during async pipeline complete; testing requires mounting the full component.
- **Why rejected**: Plugin contract demands a single coherent snapshot; multi-slice state fights that.

### Alternative 3: Direct Monaco model as source of truth

- **Description**: Treat Monaco's text model as the canonical state; sync React state on every change event.
- **Pros**: Fewest copies of the markdown string.
- **Cons**: Preview mode has no Monaco instance; SSR fallback has no Monaco; ref-based reads become async; controlled mode becomes hard.
- **Why rejected**: Lib must work in preview-only and SSR contexts, where Monaco is absent.

## Consequences

### Positive

- Single mental model for state mutations: every change goes through `dispatch`.
- `EditorAPI` is stable across re-renders, safe to capture in plugin closures.
- Controlled and uncontrolled modes share one reducer.
- Testing-friendly: render with a Context provider mock and assert on dispatched actions.

### Negative

- Context re-renders fan out to consumers; must split into `BobEditorStateContext` + `BobEditorApiContext` (the API object never changes; only state does) to avoid re-rendering plugins on every keystroke.
- Reducer grows quickly; partition action types by domain (`content/*`, `mode/*`, `selection/*`, `storage/*`).

### Risks

- Plugin `onChange` invoked synchronously after dispatch may itself dispatch, risking loops. Mitigation: enforce idempotent plugin handlers and guard with a recursion-depth counter (dev warning if >2).
- Stale closures inside plugin callbacks. Mitigation: `EditorAPI` methods always read through live refs, never close over state snapshots.

## Implementation Notes

- Reducer file: `src/core/state/reducer.ts`. Action types defined as discriminated union.
- Context split: `BobEditorStateContext` (state slice subscribed via `useSyncExternalStore` for fine-grained selection) and `BobEditorApiContext` (stable api object, simple Context).
- `EditorAPI.insertText` delegates to Monaco when present (`editor.executeEdits`), falls back to string-splicing the markdown otherwise.
- `useImperativeHandle` exposes a narrow subset of `EditorAPI` matching the public `BobEditorRef` type.
