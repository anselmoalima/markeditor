# ADR-002: Async unified pipeline rendered through `rehype-react` with debounced memoization

## Status

Accepted

## Date

2026-05-24

## Context

PRD §5.3 and §7.4 specify a `unified` pipeline composing `remark-parse`, `remark-gfm`, `remark-math`, `remark-rehype`, `rehype-katex`, `rehype-highlight`, `rehype-sanitize`, and final transform to React elements. Pipelines built with `unified` are asynchronous (`pipeline.process` returns a Promise). The component must (a) render preview within 300 ms for 10k-line documents (RF-6.1), (b) debounce re-renders by 150 ms (RF-5.3.5, configurable via `previewDebounceMs`), (c) memoize: same markdown → same React tree (RF-5.3.6), (d) surface plugin pipeline contributions (`remarkPlugins`, `rehypePlugins`, `components`), and (e) keep the preview reachable even while computation is in flight.

## Decision

Implement preview rendering as a single async pipeline factory that yields a `Processor` per-instance, composed from the static core stack plus dynamically merged plugin contributions. The final stage uses `rehype-react` with `createElement: React.createElement` and a `components` map (overridable per-plugin). Rendering is driven by a `usePreview(markdown, deps)` hook that:

1. Debounces the markdown input by `previewDebounceMs` (default 150 ms).
2. Memoizes by a stable cache key (`hash(markdown) + pluginListVersion + sanitizeSchemaVersion`).
3. Invokes `pipeline.process(markdown)` inside a `useEffect`, abortable via a generation counter — stale runs are dropped.
4. Stores the resulting `React.ReactElement` in state and renders it; while pending, the previous successful render stays mounted (no flash to spinner unless first render).

`onAfterRender` and `onBeforeParse` plugin hooks bracket the pipeline call. Pipeline errors are caught, reported via `onError`, and render a localized inline error component instead of throwing — KaTeX/Mermaid failures degrade gracefully.

## Alternatives Considered

### Alternative 1: `hast-util-to-jsx-runtime` (newer)

- **Description**: Replace `rehype-react` with `hast-util-to-jsx-runtime` (jsx-runtime aware, synchronous, smaller).
- **Pros**: Smaller bundle, faster cold render, modern jsx-runtime integration.
- **Cons**: Not specified in the PRD; ecosystem still maturing; less canonical for `unified` consumers.
- **Why rejected**: PRD §7.4 explicitly prescribes `rehype-react`; the marginal bundle win does not justify diverging from the documented architecture in v1.0. Reconsider for v1.1.

### Alternative 2: Synchronous pipeline using `processSync`

- **Description**: Use `pipeline.processSync` to avoid async state management.
- **Pros**: Simpler code path; no race conditions.
- **Cons**: Many `unified` plugins (notably `remark-math` parsers and some sanitize variants) are not safe to invoke synchronously; blocks the UI thread on large documents (>10k lines) and violates the <300 ms budget without microtask yielding.
- **Why rejected**: Async pipeline is non-negotiable for performance on large documents.

### Alternative 3: Web Worker pipeline

- **Description**: Run `unified` inside a Web Worker; main thread receives `hast` and renders.
- **Pros**: True off-main-thread render; better large-doc performance.
- **Cons**: Significant complexity (serialize/deserialize plugins, transferable AST), broken `import()` paths for plugin code shipped in worker, no SSR story.
- **Why rejected**: Out of scope for v1.0; debounce + memoization meet the 300 ms budget per PRD without it. Revisit if benchmarks fail.

## Consequences

### Positive

- Predictable async lifecycle with abort semantics; no stale renders.
- Plugin contributions compose deterministically (plugin order = registration order).
- Errors in math/mermaid stay contained.

### Negative

- More code than a synchronous pipeline; `useEffect` cleanup and generation counter must be correct.
- First render is one microtask later than synchronous alternatives.

### Risks

- React 18 strict-mode double effects fire `process` twice in dev. Mitigation: ignore results when generation counter advanced.
- Memoization cache must not retain unbounded entries across content changes. Mitigation: bounded LRU keyed on stable hash, size ≤ 4 entries (current + recent toggles).

## Implementation Notes

- Hash function: `xxhash-wasm` is too heavy; use simple FNV-1a over markdown string (32-bit) plus plugin-list signature. Adequate for cache identity.
- `pipeline.process` is invoked with the `VFile` returned by upstream; `result.result` is the `React.ReactElement` from `rehype-react`.
- `components` map merges: lib core defaults < built-in plugin components < user plugin components < user `components` prop.
- The hook returns `{ element, status: 'idle' | 'pending' | 'ready' | 'error', error }` so the preview component can render skeletons during first computation.
