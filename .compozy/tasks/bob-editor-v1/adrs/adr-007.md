# ADR-007: Prefixed plain CSS bundled to `bob-editor/styles` with `--mde-*` CSS variables as the customization contract

## Status

Accepted

## Date

2026-05-24

## Context

The library must (a) ship a single CSS file at `dist/styles.css` exported as the `./styles` subpath (PRD §6.6.2), (b) avoid leaking styles into the consumer app's global namespace, (c) allow visual customization without a recompile (PRD §5.10.3 mandates CSS variables as the contract), (d) declare `sideEffects: ["**/*.css"]` so JS bundles remain side-effect-free (PRD §6.6.2), (e) avoid CSS-in-JS runtime cost, and (f) keep the CSS readable enough for consumers to write overrides.

## Decision

Hand-write plain CSS files under `packages/bob-editor/src/styles/`. Every selector is prefixed with `.bobmd-` (e.g., `.bobmd-toolbar`, `.bobmd-toolbar__button`, `.bobmd-preview > pre`). BEM-ish naming for nested concerns. All visual tokens reference CSS custom properties on the root component element (`--mde-color-bg`, `--mde-color-fg`, `--mde-color-accent`, `--mde-font-body`, `--mde-radius-sm`, etc.) — never hard-coded values inside rule bodies.

`tsup` bundles the CSS to `dist/styles.css` via its built-in CSS pipeline. JS source files import CSS via side-effect imports only at module entry (`src/index.ts` triggers the CSS only through the explicit subpath; the default JS import is style-free). The published `package.json` `exports` map exposes `./styles` and `./styles.css` to the same file.

Theme presets (`light`, `dark`, `auto`) are sets of CSS custom property values applied to the root via `data-theme="light|dark"`. The `auto` theme uses `@media (prefers-color-scheme: dark)` to swap variable values. The `theme` prop accepting a `BobmdTheme` object writes the values inline as `style={{ '--mde-color-bg': theme.editor.background, ... }}` on the root element.

## Alternatives Considered

### Alternative 1: CSS Modules (`*.module.css`)

- **Description**: Each component imports its own scoped module; class names are hash-prefixed.
- **Pros**: Bullet-proof isolation by default; no naming discipline needed.
- **Cons**: Consumer overrides become brittle (selectors include hashes); inspect-in-devtools is messier; consumers can't write `.bobmd-toolbar { ... }` in their global CSS.
- **Why rejected**: PRD §5.10.3 documents CSS variables as the customization API; CSS Modules muddle that with hash names.

### Alternative 2: Vanilla-extract / Stylex / Panda CSS

- **Description**: Type-safe CSS-in-JS with zero runtime; tokens map to TS constants.
- **Pros**: Strong tokens story; refactor-friendly.
- **Cons**: New dependency not in PRD §15; build complexity grows; CSS output is harder to override.
- **Why rejected**: Plain CSS + variables already provides the tokens story and zero runtime cost without a new dep.

### Alternative 3: styled-components / emotion

- **Description**: Runtime CSS-in-JS.
- **Cons**: Adds runtime weight to the bundle, breaks `sideEffects: ["**/*.css"]` declaration, conflicts with SSR's expectation of static stylesheet output.
- **Why rejected**: PRD CLAUDE.md §5 explicitly prohibits styled-components / emotion inside the library.

## Consequences

### Positive

- Bundle has no CSS-in-JS runtime.
- `.bobmd-` prefix prevents collisions with consumer apps.
- CSS variables make theming declarative and reachable from any consumer stylesheet without imports.
- Theme objects via prop map cleanly to inline style on the root element.
- Plain CSS is grep-able and reviewable.

### Negative

- Naming discipline becomes a code-review item (any new selector without `.bobmd-` prefix is a bug). Mitigation: stylelint rule enforces the prefix and rejects raw color/length literals outside `:root { --mde-* }` definition blocks.
- Theming via inline style means the consumer-supplied object overrides theme presets per-instance (acceptable).

### Risks

- Specificity wars if a consumer sets `.bobmd-toolbar { color: red !important }`. Acceptable: documented as the override contract.
- Worker-mode Monaco brings its own stylesheet; that file is hosted by Monaco and out of scope.

## Implementation Notes

- `src/styles/index.css` is the entry; it `@import`s `tokens.css`, `theme-light.css`, `theme-dark.css`, then component files in `src/styles/components/`.
- `tsup.config.ts` declares a separate CSS entry so the bundled `dist/styles.css` is produced regardless of whether JS importers reference the styles.
- Stylelint config: `bobmd-prefix` plugin (custom) — enforces `.bobmd-` selector prefix and rejects bare hex/rgb in rule bodies.
- `data-theme` toggle is applied by the root component based on the `theme` prop.
